## 1. 设计原则与品牌定位

### 1.1 品牌定位

> **现代、温暖、可靠、专为留学生而生的「搭子」App。**

Pairhub 的用户是**独在异国、求学压力 + 社交饥渴**的留学生。所以设计要:

- ✅ **温暖的留白**:不堆信息,给眼睛喘息(留学场景焦虑感高,视觉要减负)
- ✅ **场景化引导**:不抽象,直接告诉用户「这个周末去图书馆 3 人可拼」
- ✅ **同辈感**:不官方,语言用「我们」「搭子」,不用「用户」「活动」
- ✅ **数据透明**:人数、状态、时间一眼可见(用户最焦虑的:「活动还开不开」)
- ❌ **不**做企业感(不是钉钉、不是飞书)
- ❌ **不**做冷淡极简(留学生场景需要情绪共鸣)
- ❌ **不**做二次元(不匹配品牌调性)
- ❌ **不**用 emoji 做主视觉(仅 create 表单 type 选择器可少量用)

### 1.2 设计铁律(4 条)

1. **「30 秒内找到搭子」驱动一切**
   - 任何流程设计都问自己:用户能在 30 秒内完成吗?
   - 列表 > 详情 > 报名 → 必须 ≤ 3 步 ≤ 30 秒
   - 任何超过 3 步的流程都重新设计

2. **CJK 优先**
   - 中文是主语言,字间距、行高、字号都按中文优化
   - `line-height: 1.55-1.65` for body,1.30 for headline
   - 字体栈:`-apple-system, 'PingFang SC', 'Microsoft YaHei', sans-serif`
   - 数字和英文降级处理(可考虑 `font-feature-settings: "tnum"` 数字等宽)

3. **状态全显化**
   - 5 态状态机(RECRUITING / FULL / STARTED / ENDED / CANCELED)必须**视觉上一眼区分**
   - 不能靠用户读字,要靠颜色 + 图标 + 位置
   - 「我报名的活动」要按状态分组,不让用户翻

4. **平台原生感 + 品牌一致**
   - 微信小程序:遵循微信设计规范(无障碍最小 88rpx)+ Pairhub 视觉
   - iOS:遵循 HIG(44pt 最小点按)+ Pairhub 视觉
   - Android:遵循 Material 3(48dp 最小点按)+ Pairhub 视觉
   - 3 端**视觉一致,但交互原生化**

### 1.3 不做的设计

- ❌ 暗黑模式炫技(浅色为主,M3 末才考虑)
- ❌ 大量动画(留学场景用户快速决策,不需要花哨过渡)
- ❌ 强个性化(算法推荐 v1.1 才上,MVP 不靠算法)
- ❌ 多巴胺配色(5 类活动色已固定,不要乱加新色)
- ❌ 复杂手势(下拉、长按、3D Touch 不做)
- ❌ 弹窗弹窗弹窗(用 bottom sheet / inline 替代 modal)

## 2. 用户画像与目标

### 2.1 4 类用户画像(同 PROJECT-OVERVIEW §1.2,这里加设计维度)

| 画像 | 描述 | 设计洞察 | 关键需求 |
|------|------|----------|----------|
| **A 学业型** | 硕士/博士新生,刚到 1-3 个月 | 焦虑感强,需安全感;重信息、轻娱乐 | 同专业优先;图书馆/咖啡厅场景;时间精准 |
| **B 运动型** | 喜欢打球/跑步,周中业余 | 高频用户,需效率;重 UI 反馈速度 | 多人拼场;场地导航;装备要求 |
| **C 娱乐型** | 想玩桌游/开黑,周中/周末 | 社交驱动,重氛围;游戏 ID 展示 | 游戏 ID 段位;语音平台;桌游规则 |
| **D 社交型** | 想认识新朋友,主题局 | 重信任建立,轻功能性 | 对方资料(学校/专业/Bio);同校 badge;评价 |

### 2.2 4 类用户的设计策略

#### A 学业型(主目标)
- **首页默认 Tab** = 「学习」分类
- 卡片强信息密度(科目 / 章节 / 地点)
- 报名人数实时显示(焦虑点:「还有没有位」)
- 详情页同专业 badge 高亮

#### B 运动型(次目标)
- 卡片显示场地 / 装备 / 强度等级
- 地图入口常驻
- 满员后推荐「附近同类」(v1.1)

#### C 娱乐型(次目标)
- 卡片显示游戏 ID / 段位 / 平台
- 群聊入口显眼(M2)
- 「老玩家 / 新手」标签

#### D 社交型(增长目标)
- 用户主页 Bio / 学校 / 专业是核心
- 「同校 / 同专业」badge
- 活动结束后「认识一下」入口(M3 评价)

### 2.3 用户目标金字塔

```
         找搭子(终极目标)
              ↑
       选活动(决策成本:低)
              ↑
    看详情 / 看对方(信息充分)
              ↑
  登录 / 收藏 / 通知(基础)
              ↑
      看见列表(打开就有内容)
```

**设计优先级**:从下到上,每一层都不能漏。

### 2.4 反向目标(用户**不**想做的事)

- ❌ 注册/填一堆资料(微信一键即可,学校/专业选填)
- ❌ 加好友 / 加群(先做活动,关系自然产生)
- ❌ 读复杂的活动规则(标题 + 1 句话 + 1 段描述足够)
- ❌ 学习产品操作(「跟微信群一样」即可)

## 3. 核心用户旅程地图(5 个)

### 3.1 旅程 1:打开 App → 找到搭子 → 报名(主旅程)

**目标**:**30 秒内**完成「打开 → 看到列表 → 报名」。

**步骤细化**:

| 步骤 | 屏幕 | 关键 UI | 心理预期 | 转化风险 |
|------|------|---------|----------|----------|
| 0. 启动 | 闪屏 → 列表 | splash 1.2s | 快速看到内容 | 启动 > 2s 流失 |
| 1. 打开列表 | 首页(index) | 6 个分类 chip + 列表 | 看到内容 | 空状态、加载慢 |
| 2. 选类目 | 同屏切换 | 横向 chip | 切换流畅 | 切换不响应 |
| 3. 浏览卡片 | 滚动 | 卡片含 5 行信息 | 快速扫读 | 信息密度太高 |
| 4. 看详情 | 详情页 | 8 个区块 | 信任建立 | 加载慢、字段缺 |
| 5. 报名 | 底部 CTA | 立即报名按钮 | 1 步完成 | 按钮置灰、报错 |
| 6. 成功 | toast + 状态切换 | 「已报名」+ 倒计时 | 心理确认 | 通知沉默 |

**转化漏斗(基于 M3 数据)**:
- 启动 → 看到列表:99%
- 列表 → 详情:30%
- 详情 → 报名:**20%**(这是要优化的核心)
- 报名 → 成功:95%

**设计优化点**:
- 列表卡片**加一个 inline 报名按钮**(详情页才能报名 → 流失 30%)
- 详情页 SignupButton 永远 sticky 在底部
- 报名成功后立即显示「群聊入口」(M2)

### 3.2 旅程 2:创建活动(主旅程之一)

**目标**:**90 秒内**完成「打开 → 4 步表单 → 发布成功」。

**步骤**:

| 步骤 | 屏幕 | 输入 | 校验 |
|------|------|------|------|
| 1. 入口 | 底部 Tab「创建」 | — | — |
| 2. 选类型 | Step 0(5 类+图标) | 1 tap | 必选 |
| 3. 填信息 | Step 1(标题/描述/人数) | 文本 | 标题 5-100、描述 10-2000、人数 2-99 |
| 4. 选时间地点 | Step 2(picker + 地图) | 时间 + 地点 | 必填 |
| 5. 预览 | Step 3(详情卡) | 确认 | — |
| 6. 提交 | toast | — | 内容安全 |
| 7. 成功 | 跳详情页 | — | — |

**设计痛点**:
- ❌ 当前 picker 是 `showActionSheet`(5 个固定选项),不灵活
- ❌ 4 步切换没有进度感
- ❌ 内容安全失败时直接 BLOCK,用户不知道原因
- ❌ 提交后跳详情页,但**用户想留在创建页继续发**

**重设计方向**:
- Stepper 顶部始终可见,显示当前步骤
- 时间 picker 用第三方(原生 `picker` mode=time,日期用 `picker` mode=date)
- 失败时弹 bottom sheet,告诉用户哪段内容违规
- 提交成功后弹 toast + 留本页,可点「继续创建」(清空表单)

### 3.3 旅程 3:打开 App → 看「我的活动」

**步骤**:
1. Tab「我的」
2. 看到头像 / 学校 / Bio
3. 看到 2 个分组:
   - **我创建的**(RECRUITING / STARTED / ENDED / CANCELED 4 段)
   - **我参加的**(同上 4 段)
4. 点活动 → 跳详情

**重设计方向**:
- 顶部加活动数量统计(「你创建了 3 个 / 参加了 12 个」)
- 状态分组用**水平 tab**,默认 RECRUITING
- 报名后超过 start_time - 1h 的活动显示「⏰ 即将开始」红色 pulse
- 评价入口(M3):ENDED 状态的活动卡片底部「去评价」按钮

### 3.4 旅程 4:管理员审批(后台)

**步骤**:
1. Admin 登录(同登录页,自动判断 role)
2. Gate 页 → Dashboard
3. 看到 2×2 KPI(Users / Activities / Signups / PushTokens)
4. 点 Activities → 待审列表
5. 看到 PENDING_REVIEW 列表(玻璃卡片)
6. 选一个 → 详情页
7. 底部 Sticky 玻璃操作栏:[Approve] [Reject with reason]
8. 拒因弹 bottom sheet,textarea + 模板
9. 提交 → 列表消失 / 红 badge 减少

**重设计方向**:
- KPI 卡片加「相对上周」的箭头
- 列表用虚拟列表(virtual scroll)支持 100+ 排队
- Reject reason 提供 3 个模板(违规链接 / 色情 / 广告)
- Approve 后弹 toast,可点「撤销」(5 秒内可撤回)

### 3.5 旅程 5:首启(新用户引导)

**目标**:**60 秒内**让新用户看到第一条活动。

**步骤**:
1. 启动 → 闪屏
2. **没登录** → 登录页(微信一键)
3. 登录成功 → 引导页(3 屏)
   - 第 1 屏:「30 分钟内找到搭子」(插画)
   - 第 2 屏:「图书馆 / 球场 / 桌游 / 开黑,任你选」(5 类活动展示)
   - 第 3 屏:「学校/专业/年级 帮你找到同类」(信任感)
4. 第 3 屏底部「开始」按钮 → 跳首页
5. 首页:首次弹「完善资料」bottom sheet(选填,可跳过)

**重设计方向**:
- 3 屏引导可跳过
- 「完善资料」可彻底跳过(注册不强制,后续在「我的」里补)
- 首启不直接弹权限申请(位置权限在使用时申请,见 `app.json` `requiredPrivateInfos`)

## 9. 交互模式

### 9.1 列表模式(首页 / 我的)

- **下拉刷新** → `onPullDownRefresh` → 重新加载第 1 页
- **触底加载** → `onReachBottom` → 加载下一页(`has_more`)
- **横向 chip 切换** → 切类目,重置列表
- **搜索** → 顶栏搜索图标 → 搜索页(关键字 + 筛选 + 排序)
- **空数据** → 居中 EmptyState

### 9.2 筛选模式(活动列表 / 搜索)

- **多选 chip** → 状态多选(RECRUITING / FULL / STARTED 复选)
- **单选 chip** → 类型单选(自习/运动/桌游/开黑/其他)
- **范围选择** → 时间范围(开始 / 结束 date picker)
- **地理选择** → 地图选点 + 半径滑块
- **应用 / 重置** → 底部 sticky 按钮

### 9.3 表单模式(创建活动 / 编辑资料)

- **Stepper** → 4 步创建,顶部显示进度
- **实时校验** → 失焦时校验 + 红字提示
- **字符计数** → 文本域右下「32/2000」
- **picker** → 调起原生 date / time / location picker
- **提交** → loading 状态 + 成功后跳详情 / 失败 toast
- **退出未保存** → 弹确认 modal(只对长表单)

### 9.4 弹窗模式

- **Toast** — 1.5s 自动消失,顶部 / 中部 / 底部可选
- **Modal** — 居中弹层,背景遮罩,**仅用于「不可恢复操作」(注销 / 取消活动)**
- **BottomSheet** — 半屏弹窗,推荐用于:
  - 表单补充(Reject reason)
  - 操作选择(长按菜单)
  - 地图全屏切换
- **ActionSheet** — 操作列表(iOS 风格),仅用于:排序选择 / 类型选择

### 9.5 导航模式

- **Tab 切换** → Tab 栏 5 个,自定义 tabbar,安全区适配
- **深链跳转** → `wx.navigateTo`,最多 5 层
- **reLaunch** → 登录成功 / 注销后
- **switchTab** → 创建成功后跳首页

### 9.6 反馈模式

- **点击反馈** → scale 0.98 + 120ms easing(`--motion-duration-fast`)
- **长按反馈** → 玻璃菜单(Admin 用)
- **加载反馈** → 按钮内置 spinner / 列表 skeleton
- **成功反馈** → toast + 状态切换(报名 → 已报名)
- **错误反馈** → toast 显示 `userMessage`(从 `ApiException.detail`)

## 10. 后台 Admin(Glass 视觉)

### 10.1 设计语言

> 现代、极简、**Apple 风格**。
> 视觉签名是**液态玻璃(Glassmorphism)** — 半透明面板 + 细描边 + 彩色渐变背景。
> 如果用户**看不到背景透过面板**,设计就失败了。

**参考**(不抄):
- iOS 18 控制中心
- visionOS 侧边栏
- Apple Weather / Maps
- macOS Sonoma 锁屏

**禁止**:
- 面板上的重渐变
- `> 0.5` 不透明度的阴影
- 单屏 > 3 个强调色
- 文案中的 emoji
- 玻璃面上的硬边框

### 10.2 5 个 Admin 页面(详 §5.10-5.14)

1. **Admin Gate** — 入口判断
2. **Activities Review Queue** — 玻璃卡片列表 + inline 按钮
3. **Activity Detail (Moderation)** — 复用消费者详情 + Sticky 玻璃操作栏
4. **Users Search** — 玻璃搜索栏 + 玻璃行
5. **Dashboard Metrics** — 2×2 玻璃 KPI 网格

### 10.3 视觉细节(关键)

- **背景**:4 色网格渐变(3 屏慢漂移,60s loop)+ 80-100px blur
- **玻璃面**:深色 `rgba(255,255,255,0.08)` / 浅色 `rgba(255,255,255,0.55)`
- **边框**:1px 细描边,深色 `rgba(255,255,255,0.18)` / 浅色 `0.65`
- **圆角**:20px(普通)/ 28px(大)
- **阴影**:`0 4px 16px rgba(0,0,0,0.18)`(浅)/ `0 8px 32px rgba(0,0,0,0.30)`(深)
- **入场**:320ms ease-out + translateY(8px → 0)
- **按下**:scale 1.0 → 0.98 over 120ms
- **背景运动**:60s loop 慢漂移,`prefers-reduced-motion` 暂停

### 10.4 深浅模式

- **默认深色模式**(深色玻璃更显气质)
- 浅色模式可切换
- 用 OS 媒体查询 `@media (prefers-color-scheme: dark)` 自动判断

### 10.5 性能

- `backdrop-filter: blur(...)` 是 GPU 密集,**只在 5 个 admin 页用**
- 列表限制 100 条,超过用虚拟滚动
- 背景动画默认关(`prefers-reduced-motion: reduce`)

## 14. 现有 UI 痛点与重设计方向

> **本节是重设计的核心**。每条痛点都来自实际代码 / M3 验证数据 / 设计 review。

### 14.1 痛点 1:列表卡片点击 vs inline 报名

**现状**:v1.0 ActivityCard 整张卡片可点,跳详情页后才能报名。

**问题**:
- 70% 报名在列表 1-3 位(数据)
- 用户需要多 1 次跳转(从列表 → 详情)
- 跳转后流失 10-20%

**重设计**:
- 卡片**右下角加 inline 报名按钮**(`flex` 占 1/3 宽)
- 整卡点击仍进详情(双层交互)
- 已报名用户:inline 按钮变「已报名 + 取消」

**实施**:
- `ActivityCard` 新增 `inline` prop(默认 true)
- `SignupButton` 改造支持 `size="sm"`(用于 inline)
- 复用 6 态设计

### 14.2 痛点 2:创建活动的 Stepper + Picker 体验

**现状**:
- `_pickDateTime` 用 `showActionSheet` 5 个固定选项(今天 14:00 等)
- 不灵活,用户只能选这 5 个
- Stepper 没有进度感

**重设计**:
- 用原生 `picker mode=date` + `mode=time`
- Stepper 顶部固定,显示当前步骤
- 「下一步」按钮 disabled 直到校验通过
- 内容安全失败时弹 bottom sheet 说明

### 14.3 痛点 3:活动详情字段多但排版乱

**现状**:
- 详情页字段:Cover、Badge、Title、人数、时间、地点、Tags、Creator、Description、Participants
- 全部堆在 scroll view 里,无视觉分组
- 关键信息(报名按钮)要滚到底部才能看到

**重设计**:
- 顶部增加 **Hero 区**(Cover + 状态 + 标题 + 人数,1 屏内)
- 报名按钮**永远 sticky 底部**
- 区块用 `--space-3xl` 48 分隔
- 每个区块有标题(可选),如「时间地点」「关于这次活动」「创建者」「参与者」

### 14.4 痛点 4:「我的」页面没统计、没分组

**现状**:
- 我创建 / 我参加 混在一起
- 无数量统计
- 状态分组不明显

**重设计**:
- 顶部统计:「你创建了 3 个,参加了 12 个」
- 状态用水平 tab:RECRUITING / STARTED / ENDED / CANCELED
- 每段独立 card 区块,带状态 badge

### 14.5 痛点 5:无障碍不达标

**现状**:
- 部分 badge 仅有颜色,无文字(违反 WCAG)
- 部分按钮 hit target < 44pt
- 状态 pill 没用 `aria-label`

**重设计**:
- 全部 badge 加 icon + 文字
- 所有 hit target ≥ 44pt / 88rpx
- 状态 pill 用 `aria-label="Status: 招募中"`
- 焦点环可见

### 14.6 痛点 6:深色模式缺失

**现状**:仅浅色模式,无 dark mode

**重设计**:
- 用 OS 媒体查询自动切换
- M3 `ColorScheme.fromSeed` 派生
- 玻璃面在深色模式更显气质

### 14.7 痛点 7:Admin 入口生硬

**现状**:Admin Gate 页直接显示「联系开发者获取权限」

**重设计**:
- 玻璃面板 + Mesh 背景(已有)
- 错误状态用品牌语气:「你不是管理员?和 @YuanshuoDu 打个招呼吧~」
- 提供「复制 SQL」按钮(已有,优化交互)

### 14.8 痛点 8:Toast 样式不统一

**现状**:用 `wx.showToast`,不可定制

**重设计**:
- 自定义 Toast 组件
- 4 类:成功 / 错误 / 警告 / 信息
- 顶部 / 中部 / 底部 3 位置
- 1.5s / 3s / 5s 3 时长

### 14.9 痛点 9:搜索是 mock

**现状**:首页搜索图标点开 → toast「搜索功能开发中」

**重设计**(M3 W12 计划):
- 搜索页(关键字 + 类型 + 时间 + 地理)
- 联想词(基于历史搜索)
- 搜索历史(可清空)

### 14.10 痛点 10:地理筛选难用

**现状**:`lat + lng + radius_km` 三参数同时传,但前端难选

**重设计**:
- 地图选点:点地图任意位置 → 自动取 lat/lng
- 半径滑块:0.5km / 1km / 2km / 5km / 10km / 20km
- 列表按距离升序

### 14.11 痛点 11:报名失败时缺诊断

**现状**:报错时只 toast `userMessage`,用户不知道为什么

**重设计**:
- 满员:toast + 「看看其他活动」CTA
- 已报名:toast + 「你已报名,跳详情」
- 活动已开始:toast + 「不能报名已开始的」+ 提示「下次早点来」
- 重复报名:toast + 「你已报名」+ 跳详情

### 14.12 痛点 12:评价入口缺失

**现状**:M3 计划,还没做

**重设计**(M3 计划):
- ENDED 活动卡片底部加「去评价」按钮
- 评价页:1-5 星 + 评论(≤ 500 字)+ 提交
- 评价历史:在用户主页

### 14.13 痛点 13:i18n 占位未铺

**现状**:仅中文

**重设计**(v1.1 计划):
- 提取所有用户可见文案到 i18n bundle
- 启动时读系统语言
- 用户可在设置中切换

### 14.14 痛点 14:加载态单一

**现状**:仅 LoadingSkeleton

**重设计**:
- 列表首次加载:skeleton
- 列表下拉刷新:顶部小 spinner
- 触底加载:底部小 spinner + 「加载中...」
- 详情页加载:cover skeleton + 文字 skeleton
- 按钮加载:按钮内置 spinner

### 14.15 痛点 15:内容安全反馈差

**现状**:失败直接 BLOCK,用户不知道为什么

**重设计**:
- 失败时弹 bottom sheet,告诉用户哪段内容违规
- 高亮违规字段
- 提供修改建议(如「请避免使用『拼单』等敏感词」)

## 17. 关键设计决策记录(ADR-like)

> 这里记录当前已定型的设计决策,供未来 review 参考。

### 17.1 D-UI-001:5 类活动色固定

**决策**:5 类活动各绑定 1 个固定色(STUDY 蓝 / SPORTS 绿 / BOARD_GAME 紫 / ONLINE_GAME 红 / OTHER 灰)。

**理由**:
- 用户识别速度提升
- 跨端视觉一致
- 未来「推荐 feed」可用类型色做分类

**不重审**直到 v2.0。

### 17.2 D-UI-002:Admin 用液态玻璃(Glass)

**决策**:Admin 5 个页面统一用 Glassmorphism。

**理由**:
- 区别于用户面(主面是「白底 + 卡片」)
- 体现后台的专业感
- Apple 风格调性

**不重审**直到 v1.1 后。

### 17.3 D-UI-003:3 端 1 套 design tokens

**决策**:颜色 / 字体 / 间距 / 圆角 / 阴影 3 端完全共用。

**理由**:
- 降低维护成本
- 视觉一致
- 平台差异只在交互层(原生按钮 / picker)

**不重审**直到 v2.0。

### 17.4 D-UI-004:CJK 优先

**决策**:中文是主语言,英文降级处理。

**理由**:
- 60% 用户是大陆学生
- 留学生场景中文阅读更顺
- i18n 准备但不是优先级

**不重审**直到 v1.1 启用 i18n。

### 17.5 D-UI-005:6 态 SignupButton

**决策**:报名按钮 6 态(default / pressed / loading / disabled / joined / danger)。

**理由**:
- 报名流程涉及业务规则多(满员 / 已报名 / 已开始 / 创建者)
- 6 态覆盖所有场景
- 用户一眼看到当前状态

**不重审**直到业务规则大改。

### 17.6 D-UI-006:状态联动必须视觉化

**决策**:5 态状态机的转换必须视觉显化(不是只文字)。

**理由**:
- 留学场景焦虑感高
- 「我报名的活动开没开」要 1 秒看到

**不重审**。

### 17.7 D-UI-007:不引入新色

**决策**:除 5 类活动色外,不引入新色。

**理由**:
- 品牌色锁定
- 重设计时复用现有 token
- 减少决策成本

**不重审**直到品牌升级。

### 17.8 D-UI-008:不堆信息

**决策**:留学场景下,信息密度上限 = 卡片 5 行 / 详情 8 区块。

**理由**:
- 用户决策成本高
- 视觉减负

**不重审**。

### 17.9 D-UI-009:列表卡片加 inline 报名

**决策**:ActivityCard 增加 inline 报名按钮。

**理由**:
- M3 数据:70% 报名在列表 1-3 位
- 减少 1 次跳转 = 提升 10-20% 转化

**实施**:v1.1 W3 落地。

### 17.10 D-UI-010:sticky 报名按钮

**决策**:活动详情页 SignupButton 永远 sticky 底部。

**理由**:
- 报名是核心 CTA
- 任何时候都能点

**实施**:v1.1 W2 落地。
