# v1.1 roadmap (M3 retrospective input)

This doc captures what the v1.1 (post-M3-launch) work looks like,
based on the M3 retrospective findings plus product inputs that
didn't fit the M3 launch window. **Living document** — the v1.0
launch is a checkpoint, not a finish line.

## v1.1 timeline

| Week | Theme | Owner |
| --- | --- | --- |
| v1.1 W1 | Stabilization: monitor M3 launch alerts, fix top 5 P0 bugs | on-call + CTO |
| v1.1 W2 | Discovery interviews: 5 of the most active users, 5 of the churned ones | PM + research |
| v1.1 W3 | v1.1 backlog grooming (the items below, scoped into 2-week sprints) | CTO + PM |
| v1.1 W4-W12 | 3 × 2-week sprints shipping the top 6 v1.1 features | full team |

The 6 candidates below are listed in **descending priority** based
on the M3 retrospective + product judgment. Re-ranked after W2
interviews + W3 backlog grooming.

## v1.1 candidates

### 1. Interest-based matching recommendation feed

**What it is**: a "recommended for you" tab on the home screen that
suggests activities the user is likely to join, based on (a) their
school + major + grade, (b) the activities their friends (mutual
follows) joined, (c) the categories they've engaged with most.

**Why now**: the v1.0 list is purely chronological (FIFO from
`startTime`). The interview transcripts from W2 will tell us if
discovery is the #1 pain point, but the data already says so:
70% of the v1.0 activity joins are 1-3 deep in the list (per the
Grafana funnel), and the median session finds 1.4 activities
before joining one. Recommendation feed should make the first
match land in the top 5.

**Scope**:
- New ranking service (`server/src/modules/recommendation/`)
- Pre-compute user embedding every 6h (cold-start: fallback to
  school+major+grade) in a batch job
- Reuse the existing activity list endpoint with a `?feed=recommended` query
- Client: new "推荐" tab on the home screen (Flutter + miniprogram)

**Effort**: 2 sprints (1 dev backend + 1 dev frontend + 0.5 PM for
ranking heuristic tuning)

### 2. 同学校 / 同专业 priority boost

**What it is**: activities from the same school or major get a
2× boost in the recommendation feed (and a "同学都在" badge in
the list). This is a special case of #1 but ships first because
it has zero cold-start cost and is a tangible trust signal.

**Why now**: the M3 launch data shows a 3.2× higher join rate when
the creator and joiner are from the same school. The user
interviews in W2 will validate whether this is a "discovery" or
"trust" thing — but either way, the lift is real.

**Scope**:
- Server: enrich `GET /api/v1/activities` response with a
  `sameSchool: boolean` and `sameMajor: boolean` per activity
- Flutter + minipgm: render the "同学" badge
- Recommendation feed: factor the same-school boost in

**Effort**: 0.5 sprint (server change + 2 client pages + 1 PM week)

### 3. 拼车出行 (carpool to events)

**What it is**: when joining an activity, the joiner can also opt
into a "carpool" seat. The activity creator can mark the activity
as "carpool-friendly" (typically any off-campus venue). Drivers
can post their car; riders can request a seat. M3 v1.0 doesn't
have this — the interview transcripts from W2 will tell us if
the off-campus event volume is high enough to justify the
investment.

**Why now**: the M3 activity types are STUDY / SPORTS / BOARD_GAME /
ONLINE_GAME / OTHER. Off-campus events are mostly SPORTS +
"OTHER" (hikes, day trips). If those categories grow past 30%
of activity volume, carpool is the natural next feature.

**Scope** (subject to interview validation):
- Server: `Carpool` model, `CarpoolSeat` model, REST endpoints
- Flutter: carpool section on the activity detail page
- Minipgm: same
- Background job: weekly driver reliability score (5-star avg
  per driver from completed trips)
- No payment / escrow in v1.1 — that's v2.0 territory (see "Out of
  scope" below)

**Effort**: 2.5 sprints

### 4. 多语言 (i18n) — Chinese + English

**What it is**: the app ships with both 中文 and English UI strings,
chosen at first launch and editable in settings. Server stores
the user preference; client picks the right bundle.

**Why now**: the user base is roughly 60% Mainland Chinese
students (who use 中文 natively) + 40% international (English
only). The current all-Chinese UI is a discoverability blocker
for the 40%. The fix is small: ~200 strings, 2 bundles, a `locale`
field on the User model.

**Scope**:
- Server: add `User.locale` field (zh-CN | en-US, default zh-CN)
- Minipgm: wxs bundles + `wx.getSystemInfoSync().language` as default
- Flutter: intl + ARB bundles
- Settings: language picker

**Effort**: 1 sprint

### 5. 支付 / 押金 (payment + deposit)

**What it is**: support paid events (host sets a price) + deposits
(hold the joiner's card, refund on cancel, forfeit on no-show).
Required for any event that's actually monetized (tickets, classes).

**Why now**: the v1.0 interview transcripts will reveal the demand.
Payment integration is heavy (Stripe + WeChat Pay + refund
handling + tax compliance) and we want to be deliberate about
adding it. Likely v1.1 W6-W10 (third sprint), conditional on
W2-W3 demand validation.

**Scope** (subject to interview validation):
- Server: `PaymentIntent` model, webhook handlers (Stripe +
  WeChat Pay), refund flow
- Admin: payment dashboard (refund queue, chargebacks, tax report)
- Flutter: Stripe SDK + WeChat Pay SDK
- Minipgm: WeChat Pay SDK
- Background job: nightly reconciliation between Stripe / WeChat
  Pay / internal ledger

**Effort**: 3 sprints + dedicated launch week. This is the biggest
single feature on the v1.1 list.

### 6. Mini-game / group ice-breakers

**What it is**: 2-3 lightweight in-app games that help strangers in
a group activity break the ice ("Two Truths and a Lie", "Would
You Rather", "Quick Story Round"). Plays a round before the
activity starts in the chat.

**Why now**: the v1.0 chat is utilitarian. The user interviews
from W2 will tell us if the social-bonding blocker is real (vs.
e.g. just "we need a payment system"). If real, this is the
highest-leverage engagement play.

**Scope** (subject to interview validation):
- Client-only — no server changes
- Minipgm: 2 small mini-game components, ~600 lines
- Flutter: 2 small games, ~800 lines
- No leaderboards, no friends, no profile integration in v1.1

**Effort**: 1 sprint

## Out of scope for v1.1 (deferred to v2.0+)

| Feature | Why deferred |
| --- | --- |
| Live streaming (TUDLive) | Engineering weight (WebRTC) doesn't match M3 launch volume |
| In-app payments (Stripe Connect, WeChat Pay merchant onboarding) | Compliance + tax + KYC is a quarter's work by itself |
| AI / LLM features (auto-translate, group summary, etc.) | Costs are unpredictable at our scale; wait for v1.1+ volume data |
| Group buy / discount codes | Marketing channel infrastructure we don't have yet |
| Native iOS / Android rebuild (skip Flutter) | Flutter is the right call for v1.x; revisit when DAU > 50k |

## Dependencies on M3 retro findings

The 6 candidates above are *not* ranked by gut. They depend on the
answers to 4 questions from the M3 retrospective:

1. **What % of joined activities are off-campus?** (drives #3 carpool)
2. **What's the most common reason users don't join the first activity
   they look at?** (drives #1 recommendation, #2 same-school)
3. **What % of users hit a payment-related dead-end?** (drives #5 payment)
4. **What does the social-bonding funnel look like — 1st-message
   latency, "second-event" retention, "made-a-friend" survey?** (drives #6 game)

If the M3 retro reveals a different #1 pain point (e.g.
notifications, on-boarding, content moderation at scale), the
v1.1 sprint plan is re-prioritized before W3.

## Risks to the v1.1 timeline

- **Engineering hiring** — we need 1 backend + 1 frontend + 0.5 PM
  for the v1.1 plan. Currently the team is at CTO + 1 backend +
  1 Flutter + 0 minipgm. If we can't hire by W4, the v1.1 plan
  slips to a 14-week timeline.
- **Payment compliance** — see #5 above. If the demand is real but
  we can't ship Stripe in 2 sprints, we drop #5 from v1.1 entirely
  and bank the demand signal for v2.0.
- **Recommendation quality** — a bad recommendation feed is worse
  than no feed (it amplifies the "your feed is bad" complaint).
  We'll A/B test before fully shipping #1.

## Open questions for W2 user interviews

1. What's the first thing you look for when opening the app?
2. How many activities do you see before joining one?
3. What would make you join MORE activities than you currently do?
4. Have you ever paid to attend an event like this? If yes, what
   was the rough price range?
5. Do you use the in-app chat? If yes, what's the longest
   conversation you've had there?
6. Would you use a "推荐" tab if it showed you activities from
   people at your school / major / with shared interests?
