# OpenAPI / Codegen Architecture

The Pairhub API surface is authored **once** in TypeScript and shipped
to three clients (browser docs, WeChat minip, Flutter app) through a
single-source-of-truth pipeline. This page is the architecture map —
file-by-file, command-by-command, with a section on how to add a new
endpoint without breaking any client.

## TL;DR

```
server/src/lib/openapi-spec.ts        (typed, hand-curated)
  |
  |  pnpm -C server run openapi:build
  v
docs/api/openapi.json                 (110 KB, 14 paths, 6 schemas)
  |
  |  pnpm -C miniprogram run openapi:codegen     (openapi-typescript)
  v
miniprogram/types/openapi.ts          (95 KB, immutable + enum)
  |
  |  node tools/codegen-dart.mjs                 (pure Node, 0 deps)
  v
app/lib/api/openapi/*.dart            (6 files + barrel, ~850 lines)
```

The pipeline is **deterministic + committed** — every artifact is
regenerated from the spec and checked in, so clients can clone the repo
and see the contract without booting the server.

## Files

### Authoring (server)

- **`server/src/lib/openapi-spec.ts`** — the source of truth. A typed
  `OpenAPIV3.Document` covering the 18 v1 endpoints from
  `docs/api/v1.md`. Three reusable helpers keep the per-path payload
  DRY:
    - `successEnvelope(data)` — wraps a data shape in `{ data, meta }`
    - `errorResponse(code, message)` — emits a 4xx/5xx `ResponseObject`
    - `jsonResponse(description, schema, status)` — emits a 2xx
      `ResponseObject`
- **`server/src/lib/openapi.ts`** — the `fastify-plugin` that registers
  `@fastify/swagger` in **static** mode and `@fastify/swagger-ui` under
  `/api/v1/docs`. The raw spec is auto-served at `/api/v1/docs/json`.
- **`server/src/lib/app.ts`** — calls `app.register(openApiPlugin)`
  before any cross-cutting plugin, so docs are live by the time the
  first request hits.
- **`server/scripts/build-openapi-json.ts`** — the CLI that imports
  the typed spec and serializes it. Invoked via
  `pnpm -C server run openapi:build`.
- **`server/tests/openapi.test.ts`** — 4 cases:
    - `GET /api/v1/docs` serves the swagger UI HTML
    - `GET /api/v1/docs/json` returns 200 + the spec
    - the spec contains the 14 v1 paths we care about
    - public vs JWT endpoints carry the right `security` block
- **`server/tests/openapi-json-sync.test.ts`** — re-serializes the spec
  in-process and asserts the on-disk `docs/api/openapi.json` matches
  byte-for-byte. Catches "edited the spec, forgot to run
  `openapi:build`".
- **`server/package.json`** — adds the `openapi:build` script.

### JSON artifact (root)

- **`docs/api/openapi.json`** — generated; **do not hand-edit**. The
  author/regen contract: edit the spec, run `openapi:build`, commit
  both.

### TS codegen (minip)

- **`miniprogram/types/openapi.ts`** — generated by
  `openapi-typescript` (`--enum --immutable`). The whole surface is
  one exported `paths` interface with inline success/failure shapes,
  plus a `components` map.
- **`miniprogram/package.json`** — adds the `openapi:codegen` script
  and `openapi-typescript` as a devDep.

### Dart codegen (app)

- **`tools/codegen-dart.mjs`** — a 400-line pure-Node ESM script. No
  external deps. Walks `components.schemas` and emits one Dart
  `const class` per schema + a barrel. Reusable helpers for enum
  emission, JSON parsing, JSON emission, nullable handling.
- **`tools/README.md`** — the operator's manual for the script.
- **`app/lib/api/openapi/{user,creator,activity,signup,meta,pagination}.dart`**
  — generated DTOs.
- **`app/lib/api/openapi/openapi.dart`** — generated barrel
  (`export 'user.dart'; export 'creator.dart'; ...`).

## Why static spec (not dynamic)

`@fastify/swagger` has two modes:
- **`mode: 'static'`** (used here) — you pass the full spec object; the
  server serves it verbatim.
- **`mode: 'dynamic'`** — the plugin walks registered routes and
  builds the spec from each route's `routeConfig.schema` (the body,
  params, querystring, response JSON Schema).

The Pairhub production handlers parse zod schemas **in-handler**,
e.g.:

```ts
const parsed = updateMeBodySchema.safeParse(req.body);
if (!parsed.success) throw new ValidationError({ issues: parsed.error.flatten() });
```

There is no `schema: { body: ... }` on the route, so the dynamic
generator has nothing to walk. Static mode is the honest choice.

### Future: switch to dynamic

The follow-up refactor moves each handler to
`fastify-type-provider-zod`, attaches zod schemas to routes, and
switches the swagger plugin to dynamic mode. After that the spec
auto-derives from the zod schemas and the hand-curated `openapi-spec.ts`
goes away. Tracking in this doc; not yet scheduled.

## Why a custom Dart codegen (not `openapi-generator-cli`)

Surveyed options:
- `openapi-generator-cli` (Java): 200 MB, generates clients for
  every language. The `dart-dio` output is 40+ files of API-client
  boilerplate the team would fight against.
- `openapi_dart_codegen` (Dart-native): emits a full API client
  that doesn't match the hand-written dio client.
- **Hand-rolled Node (chosen)**: 400 lines, zero deps, emits exactly
  the shape the team wants (Dart 3 enums + `Wire` extension,
  `fromJson` factory + `toJson` map, immutable `const` constructor).
  Auditable.

The generator handles the v1 spec's feature set today: object schemas
with `required` + `properties`, string/integer/number/boolean
primitives, enums (real Dart enums + `Wire` for wire round-trip),
`format: 'date-time'` → `DateTime`, `nullable: true` → `T?`, `array` +
`items` → `List<T>`, `$ref` → inlined as the named type. Unknown
shapes become `Object?` with a stderr warning so they don't get
silently dropped.

## How to add a new endpoint

1. **Add the path** to `server/src/lib/openapi-spec.ts` with a
   `successEnvelope` / `errorResponse` body, following the existing
   18 endpoints for shape.
2. **Run the build chain**:

   ```bash
   pnpm -C server run openapi:build           # spec -> openapi.json
   pnpm -C miniprogram run openapi:codegen    # openapi.json -> openapi.ts
   node tools/codegen-dart.mjs                # openapi.json -> *.dart
   ```
3. **Commit the four files**: the spec change, the JSON, the TS
   types, and the Dart types.
4. **The drift test** (`server/tests/openapi-json-sync.test.ts`)
   will fail CI if step 2 is skipped.

The minip + app teams don't need to take any additional action —
their next `git pull` includes the new types.

## Drift story

- **Spec change without `openapi:build`** → caught by
  `openapi-json-sync.test.ts` (server's `pnpm verify`).
- **`openapi:build` without TS/Dart regen** → caught by review
  (the diff for `docs/api/openapi.json` should always be accompanied
  by a diff for `miniprogram/types/openapi.ts` and
  `app/lib/api/openapi/*.dart`).
- **Hand-edit of a generated file** → caught by review (the file's
  header comment says `GENERATED ... do not edit`).

A future CI workflow can run the full chain in `pull_request` and
post a comment listing any drift, but it's not yet wired.

## Versioning

The spec is a 3.0.3 document, served as the v1 API per
`docs/api/v1.md §1.1`. New endpoints or breaking changes go to a v2
spec under `docs/api/v2.md` (not yet started). The OpenAPI
`info.version` field tracks the spec version separately from any
specific app build.
