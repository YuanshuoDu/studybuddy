# Pairhub — 交付评审规范（v1.0）

> **核心原则：不交出"我做了"，只交出"我做了什么 + 怎么证明"。**

## 1. 三件套证据

每条交付必须附：

| 类型 | 适用场景 | 提交方式 |
|------|----------|----------|
| 🖼 截图/录屏 | UI、配置、看板、运行结果 | PR 评论 + 群内贴图 |
| 📋 命令输出 | 启动、CI、测试、迁移 | PR 评论贴关键行 |
| 🔗 可访问链接 | 文档、issue、PR、看板、预览页 | 群内贴 URL |

## 2. 各任务类型的证据清单

### 2.1 后端 API 改动
- [ ] 本地 `pnpm test` 全绿截图
- [ ] `pnpm lint` 通过截图
- [ ] CI Actions 跑绿截图
- [ ] 至少 1 个新/改 endpoint 的真实请求截图：
  ```
  Request:  curl -X POST http://localhost:3000/api/activities -d '...'
  Response: 201 { "id": "..." }
  ```
- [ ] 如有 DB 变更：`pnpm prisma migrate dev` 输出 + DB 工具截图

### 2.2 小程序前端改动
- [ ] 微信开发者工具运行截图（关键页面 2-3 张）
- [ ] 真机预览截图（首页 + 核心流程 1 屏）
- [ ] 录屏 GIF/MP4：核心交互（创建活动/报名/列表筛选）≤ 30 秒
- [ ] 控制台无 error 截图

### 2.3 文档输出
- [ ] GitHub 渲染后的预览截图
- [ ] 目录树截图
- [ ] 内部链接全部可点开（截图证明）

### 2.4 预研/调研类
- [ ] 每个结论附官方/可信源截图
- [ ] 引用 URL（带圈选）
- [ ] 至少 2 个备选方案的对比表

### 2.5 脚本/工程类
- [ ] 本地实际跑一遍的最终输出截图
- [ ] 失败 case 也保留（证明处理过边界）

## 3. 评审流程

每个 PR 触发：
1. **自动检查** — CI（lint/test/build）必须全绿
2. **Owner Review** — CODEOWNERS 指派的 reviewer 检查**证据齐全度**
3. **证据打勾** — Reviewer 按本文 §2 清单逐项打勾，缺一项打回
4. **CTO 抽查** — @Oracle Hermes 抽 20% PR 做深度 review

**没有证据的 PR 一律打回，不接受"我本地跑过了"。**

## 4. 进度汇报中的证据

每日 22:00 进度汇报必须包含：
- 当日完成项（每项附 PR 链接 + 截图）
- 进行中项（附 issue 链接 + 看板截图）
- 阻塞项（附错误截图或日志）

## 5. 截图归档规则

所有截图统一存：
```
docs/screenshots/
├── issue-12/            # 对应 issue 编号
│   ├── 01-首页.png
│   ├── 02-创建活动.png
│   └── 03-后端日志.png
├── issue-15/
└── daily-2026-06-01/    # 每日汇报证据
```

文件命名规范：`序号-描述.png`

## 6. 反例（不允许出现）

❌ "已完成" — 无截图/日志/链接
❌ "本地跑过了" — 无运行输出
❌ "应该没问题" — 无测试截图
❌ "参考官方文档" — 无文档截图
❌ "明天再截图" — 当日必须出
